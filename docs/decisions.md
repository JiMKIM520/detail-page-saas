# DetailAI 의사결정 기록

> 프로젝트 진행 중 내려진 주요 기술/디자인/비즈니스 결정과 그 근거.

---

## 2026-04-03: 프로젝트 시작 ~ 모델 전환

### D-001: Imagen → Gemini 전환
- **결정**: Vertex AI Imagen 3 → Google AI Studio Gemini (Nano Banana) 전환
- **근거**: Imagen은 한글 텍스트 불가, Subject Reference API 400 에러, 2026.06 deprecated
- **대안 검토**: Imagen 유지 + 영문 프롬프트 → 사용자가 "기술 문제 시 방향 변경 금지, 해결하라" 피드백
- **결과**: Gemini 3 Pro / NB2 / Standard 3종 모델 체계 확정

### D-002: SDK 선택 — @google/genai
- **결정**: `@google/genai` SDK 사용 (REST 직접 호출 대신)
- **근거**: SDK가 더 안정적, AI Studio/Vertex AI 양쪽 지원 (인증 1줄만 변경)
- **영향**: Vertex AI 전환 시 코드 변경 최소화

### D-003: 비용 예산 — 건당 10,000원
- **결정**: 상세페이지 1건당 ~10,000원 비용 허용
- **근거**: 사용자 직접 결정. 퀄리티 최우선.
- **실제 비용**: 약 2,850~3,200원/건 (예산의 28~32%)

## 2026-04-04: 디자인 퀄리티 실험

### D-004: 최종 산출물은 PDF가 주
- **결정**: PDF가 주 산출물 (디자이너 수정용), 기업에는 PNG
- **근거**: 디자이너가 PDF로 수정 가능 → 이미지 추출 → 기업 제공
- **변경 전**: 섹션별 PNG가 주, PDF가 부

### D-005: 가로 860px 고정, 세로 최소 10,000px
- **결정**: 플랫폼별 분기 불필요. 860px 통일, 세로 자유
- **근거**: 사용자 결정. 대부분 플랫폼이 800~900px

### D-006: 스타일링샷 — 스튜디오 배경만
- **결정**: 카페, 주방 등 실제 장소 배경 금지. 스튜디오 세팅만.
- **근거**: 실제 스타일링 촬영은 스튜디오. 장소 배경 넣으면 어색해짐.

### D-007: 제품 보존 프롬프트 방식
- **결정**: "Subject 유지 + 배경만 재생성" 프롬프트
- **근거**: Gemini가 누끼컷을 재해석하면 가짜 패키지 생성, 버터 합성 어색 등 문제 발생
- **프롬프트 핵심**: "빵의 형태, 크러스트 색상, 소금 결정, 질감을 원본 그대로 보존. 빵을 새로 그리거나 재해석하지 마세요."

### D-008: AI 느낌 감소 — 카메라 렌즈 용어 + 필름 그레인
- **결정**: 영문 프롬프트에 "Canon 5D Mark IV, 85mm f/2.8, Kodak Portra 400 film grain" 등 포함
- **근거**: 리서치 결과 카메라 용어가 AI 이미지 사실성을 크게 향상
- **금지 단어**: `perfect`, `clean`, `symmetrical`, `4K`, `hyperrealistic`

### D-009: 히어로 타이포 — B방식 (Gemini가 배경에 타이포 추가)
- **결정**: Step1 배경+제품(텍스트없이) → Step2 Gemini가 같은 이미지에 타이포 직접 추가
- **대안 검토**: 
  - A방식 (타이포 따로 생성 → Sharp 합성) → 합성이 어색
  - CSS 오버레이 → "폰트 올려놓은 느낌"으로 디자인 아님
- **근거**: Gemini Pro가 배경을 보고 어울리는 타이포를 직접 디자인하면 자연스러움

## 2026-04-04~05: 디자인 방향 전환

### D-010: 이모지 아이콘 절대 금지
- **결정**: `🌾🚫📦🔢` 등 이모지 사용 금지. 프로 SVG 라이브러리(Phosphor, Lucide 등) 사용.
- **근거**: 이모지는 OS마다 다르고 디자인 통일성 없음. "전문 디자이너 느낌" 불가.

### D-011: 템플릿/프리셋 금지 — 레퍼런스 기반 커스텀
- **결정**: 제품마다 레퍼런스를 검색해서 매번 새로 디자인
- **근거**: "200개를 다 같은 템플릿으로 만들면 결국 템플릿 느낌" — 사용자 결정
- **영향**: Art Director가 매번 웹 검색 + 스타일 가이드 생성

### D-012: Stitch MCP — 상세페이지에 부적합
- **결정**: Stitch는 상세페이지 제작에 사용하지 않음
- **근거**: Stitch는 웹 UI/앱 화면 생성 도구. 아무리 프롬프트를 바꿔도 "홈페이지" 형태로 나옴.
- **대안**: Stitch 디자인 시스템(DESIGN.md)은 참고용으로만

### D-013: Gemini 단독 디자인 → 하이브리드 방식
- **결정**: Gemini 이미지 생성 + HTML/CSS 정보 섹션 혼합
- **근거**: 
  - Gemini 단독 → "디자인의 사진"만 나옴, 촌스러움
  - HTML 단독 → 깔끔하지만 아마추어, "디자인" 아님
  - 하이브리드 → 임팩트 섹션은 Gemini, 정보 섹션은 HTML

### D-014: 프롬프트/페르소나 — 영어 중심
- **결정**: 모든 AI 프롬프트와 페르소나를 영어로 작성
- **근거**: LLM이 영어를 더 정확하게 이해. 특히 촬영/디자인 기술 용어.
- **예외**: 렌더링할 한글 텍스트, 한국 법규/고유명사는 원어 유지

## 2026-04-05: 에이전트 구조 확정

### D-015: 6-에이전트 구조 (PM + Art Director + 실행 5 + QA + Validator)
- **결정**: 단일 에이전트 아닌 멀티 에이전트 오케스트레이션
- **근거**: 통일성은 "스타일 가이드 JSON 계약서"로 보장. 병렬 실행으로 속도.
- **에이전트**: PM, Art Director, 아이콘, 스타일링샷, 레이어이미지, CopyWriter, HTML, QA, Validator

### D-016: Art Director가 styling-shots-prompts.json도 생성
- **결정**: 스타일링샷 프롬프트를 하드코딩이 아닌 Art Director가 제품별 동적 생성
- **근거**: 기존 gen_styling_shots.js가 "소금빵 전용" 하드코딩이었음

### D-017: 아이콘 라이브러리 — 카테고리별 매핑
- **결정**: 제품 톤에 따라 다른 아이콘 라이브러리 선별
- **매핑**: 식품=Phosphor duotone, 프리미엄=Lucide, 테크=Tabler, 친근=Iconoir
- **근거**: 아이콘도 브랜드/제품 톤앤매너에 맞춰야 함

### D-018: 웹앱 방식 유지 (데스크톱앱 검토 후 기각)
- **결정**: Next.js 웹앱 유지. Claude Code 데스크톱앱 방식 기각.
- **근거**: Claude API 비용이 전체의 2%밖에 안 됨. 절감 효과 미미. 웹앱이 배포/관리 심플.

## 2026-04-05~06: 운영 플로우

### D-019: 기업 자가 입력 + 관리자 검수
- **결정**: 기업이 웹앱에서 직접 정보 입력 → 관리자가 검수 후 파이프라인 시작
- **변경 전**: 관리자가 기업 정보를 입력하는 방식

### D-020: 기업 입력 필드 확장 (7단계)
- **결정**: 기본정보, 브랜드아이덴티티, 디자인선호도, 레퍼런스, 필수포함사항, 금지사항, 자료업로드
- **핵심 추가**: 필수 포함 사항 (인증/수상/필수문구), 금지사항, 스타일 방향 선택

### D-021: 수정 요청 — 관리자 판단 후 에이전트 재실행 or 직접 수정
- **결정**: 기업 코멘트 → 관리자가 판단 → 에이전트 재실행 적절하면 재실행, 아니면 직접 수정
- **근거**: 간단한 수정은 관리자 직접이 빠름

### D-022: 진행 알림 — 관리자는 대시보드, 기업은 최소한
- **결정**: 관리자에게 실시간 대시보드. 기업에게는 1차 결과물 완료 알림 1회만. 최종 납품은 관리자가 이메일로 직접 발송.

### D-023: 스크립트 중간 검수 — 관리자 직접 수정 가능
- **결정**: Script Writer 완료 → 관리자 검수. 관리자가 승인/재실행 요청/직접 수정 선택 가능.

## 보류/후순위 결정

### D-P01: 영상 생성 (Veo)
- **상태**: 기각 (AI 생성) / 대안: 관리자 실촬영 + AI 후처리
- **테스트 (2026-04-08)**: Veo 2, Veo 3, Veo 3 Fast 모두 테스트 → 전부 AI 티 심함. 식품 단면/텍스처 모션에서 부자연스러움 뚜렷.
- **결론**: AI 영상 생성은 현 시점 불가. 영상이 필요하면 관리자가 누끼컷 촬영 시 3초 영상 촬영 → AI가 후처리(크롭, 컬러 그레이딩, GIF 변환)하는 하이브리드 방식.
- **대안 리서치**: Hailuo-02, Runway Gen-4.5, Kling 3.0 — 추후 모델 발전 시 재검토

### D-P02: Google Stitch MCP
- **상태**: 상세페이지에는 미사용
- **근거**: D-012 참조. 웹 UI 도구라서 상세페이지 부적합.

### D-P03: GPT-4o 이미지 생성
- **상태**: 미채택
- **근거**: Gemini Pro가 이미지 품질에서 우수. GPT-4o는 "그래픽 디자인 레이아웃"에서 장점이 있다는 리서치 있었으나 프롬프트 문제일 수 있음.
