# 플랫폼 워크플로 MVP 계획 (작업프로세스 PDF → 구현 매핑)

기준: 클라이언트 작업프로세스 문서(작업프로세스_AI상세페이지.pdf, 2026-07-16 수령) + 코드 정찰(4방향).
목표: **8월 첫째 주 서비스 시작** — 오늘(7/16)부터 실질 2.5주.
원칙: 기존 자산 위에 얹는다(재작성 금지). 상태·태그·자동화는 전부 시스템 계층(수동 운영 금지).

## 0. 기존 자산 ↔ 신규 요구 총괄

| PDF 요구 | 현황 | 판정 |
|---|---|---|
| 역할 3종(관리자/디자이너/기업) | admin·designer·client 로그인/레이아웃 분리 완비 | **재사용** |
| 상태 흐름 | status-machine 14종 + TRANSITIONS | **확장** (노드 5개 추가) |
| 관리자 현황판 | /dashboard 5컬럼 칸반 + 필터 + 미배정 배지 | **재사용+확장** |
| 관리자 화면1(의뢰서→AI초안) / 화면2(1차시안→완료) | flat 리스트(planner·designer)만 | **칸반 2종 신규** |
| 디자이너 작업화면(배정→완료 4열) | photography·designer 분리 리스트 | **통합 칸반 신규** |

| 포토그래퍼(시트 리스트업→드라이브→자동반영) | 연동 코드 전무, 20MiB 업로드 제한 | **단계 축소(MVP=수동 업로드)** |
| PSD/JPG 원본 전달 | 대용량 업로드 경로 없음(PSD MIME 거부) | **신규 버킷+UI** |
| 수정 최대 2회 | usage_limit·RevisionGuide 2회 존재 | **재사용**(projects.revision_count 신설로 명시화) |

## 1. 상태 머신 확장 (S1)

기존 14종 유지 + 노드 추가(기존 화면 호환 — 추가만, 개명 없음):

```
[신규] invited(메일전송) → intake_submitted(의뢰서 작성)
       ↳ 태그: 보완·재작성·수령/미수령 (의뢰서 게이트)
기존   script_* / design_planning / prompt_ready / photo_uploaded  (스크립트·스타일링 게이트)
기존   design_generating(AI초안) — 디자이너 권한, AI 재작업 Max 2회 카운트
[신규] designer_working(작업중 — 디자이너 업글 수작업)
[신규] draft_submitted(1차시안 — 디자이너 업로드 완료, 관리자 검수 대기)
기존   design_review(기업 확인) — client_viewed_at 기록, 7일 무회신 = 승인 간주
[신규] revision_1 / revision_2(기업 수정) — projects.revision_count로 한도 강제
기존   design_approved → delivered(완료 — 자동 발송)
```

- DB: `projects.status` CHECK 제약 신설(전이표와 단일 출처 동기화), `due_date`(D-day 기준),
  `product_received_at`, `intake_approved_at`(둘 다 차면 due_date = +14일 자동),
  `revision_count`, `client_viewed_at`, `client_delivered_at`.
- 태그: `projects.tags JSONB`(경량) — { hold, revise(보완), revision_n, reviewing(검토중), rewrite(재작성) }.
  정렬 규칙은 쿼리 계층에서: 태그 보유 카드가 해당 칸 최상단 점유, 이후 도착순.
- 배정 이력: `project_assignments`(project_id, role, staff_id, assigned_by, at) — 덮어쓰기 유실 방지.

## 2. 화면 (S2)

1. **관리자 현황판**(기존 /dashboard 확장): 컬럼을 PDF 화면1 기준으로 재정의(메일전송/작업의뢰서/
   스크립트·스타일링/AI초안), 태그 배지·D-day 표시, 카드 최상단 점유 정렬.
2. **관리자 시안 보드**(신규 /dashboard/review): 1차시안/기업 수정요청/완료 3열 + 디자이너 이름 배지
   (PDF 화면2). 보완·수정N 태그 및 만족/불만족 액션 버튼(전이+메일 트리거).
3. **디자이너 보드**(신규 /my-work): 작업배정/작업중/1차시안/완료 4열, 본인(designer_id) 카드만.
   1차시안 업로드(PSD·JPG) + 제출 → draft_submitted 전이. Hold·보완 최상단 점유.
4. **기업 화면 보완**: 수정요청 CTA 명확화(코멘트→수정요청 제출 버튼), 수정 회차 표시,
   AI 활용 고지·동의 체크박스(의뢰서 — 회의 결정 사항).
5. **디자이너별 현황 대시보드**(관리자 열람): /dashboard 필터 확장으로 흡수(별도 화면 아님).
- 공용: `KanbanCard` 컴포넌트 추출(dashboard 인라인 JSX → 태그·담당자·D-day 포함).
- 배정: 칸반 인라인 select(기존 /api/projects/[id]/assign 재사용). 드래그&드롭은 후순위.

## 3. 자동화 (S3)

| 트리거 | 액션 | 구현 |
|---|---|---|
| 관리자 '초대' 액션 | 작성 링크 메일+문자, status=invited | 신규 API + Resend/SMS |
| 의뢰서 1주 미작성 | 문자 재발송 | **Vercel Cron(일 1회)** |
| 의뢰서 보완 판정 | 보완 메일/문자 자동 발송, 태그 | 기존 검수 UI에 액션 추가 |
| 제품도착+의뢰서 만족 | due_date 자동 산정(D-14) | 서버 로직 |
| 1차시안 관리자 만족 | HTML 열람 링크 메일/문자 | 기존 send-draft 확장 |
| 기업 7일 무회신 | 승인 간주 → 완료 플로우 + 안내 메일 | Vercel Cron |
| 완료 | PSD·JPG·스타일링 자동 발송(다운로드 링크) | 기존 delivered 메일 확장 |
| 2차 보완 | 자동화 금지 — '유선 필요' 태그만 | 규칙 |

- 메일: Resend 유지(RESEND_API_KEY + 발신 도메인 DNS 설정 필요 — **사용자 액션**).
- 문자: 수단 미정(**결정 필요** — 카카오 알림톡(솔라피 경유) vs SMS. 발신번호/채널 심사 리드타임
  1~2주이므로 **이번 주 내 결정·신청 필수**).
- 크론: vercel.json `crons` 2개(리마인드·무회신 간주) + 안전장치(중복 발송 방지 발송 로그 테이블 `notifications`).

## 4. 파일 플로우 (S3)

- 신규 버킷 `deliverables`(private): 디자이너 PSD·JPG 업로드(용량 상한 상향, MIME에 PSD 허용),
  경로 `projects/{id}/v{n}/`. 완료 메일은 만료형 signed URL.
- HTML 시안 서빙을 로컬 output/ 의존에서 **Storage 단일 경로로 전환**(Vercel ephemeral 리스크 제거).
- 이미지 보호 강화(후순위 P2): preview signed URL 단기 만료.
- 포토그래퍼(MVP 축소): 스태프 업로드 UI(신규 버킷)로 시작. 구글시트 리스트업·드라이브 자동반영은
  **서비스 시작 후 P2**(연동 계정·권한 결정 대기).

## 6. 결정 사항 (2026-07-20 확정)

1. **문자 수단**: 문자/카카오 알림톡 — 기술 문제 없음(솔라피 등 중계 API). 알림톡은 카카오 채널+
   발신프로필 심사 1~2주 + 템플릿 사전 승인 필요, SMS는 발신번호 사전등록 필요 → 즉시 신청 권고.
   실패 시 SMS 대체발송 옵션 표준 사용.
2. **메일**: 별도 도메인 없이 Gmail 사용 확정 — Resend(도메인 인증 필수) 대신 **Gmail SMTP**(앱 비밀번호)로
   전환. 제약 고지: 일 발송 한도(일반 500/Workspace 2,000), 스팸함 분류 위험, 발신자 신뢰도 —
   MVP 규모(200사)에서는 수용 가능, 성장 시 도메인 전환 권고. lib/email/send.ts를 프로바이더
   추상화(Resend→Gmail SMTP)로 변경.
3. **기업 로그인**: PDF대로 **사업자명 + 사업자번호 뒷 5자리**로 변경 확정. 메일 수신 주소는
   의뢰서에서 별도 수집(자동 발송 대상).

## 6-1. 잔여 결정 (S3 전까지)

1. **PSD 원본 전달 방식**: 플랫폼 다운로드(signed URL) vs 구글 드라이브 링크.
2. **포토그래퍼 MVP 범위**: 수동 업로드 시작 동의 여부(시트·드라이브 자동화는 P2).
3. AI 활용 고지 문구·동의 체크 위치(의뢰서) — 클라이언트 법무 확인 대기.
4. **알림톡 심사 신청**(사용자 액션): 카카오 채널·발신프로필·템플릿 — 리드타임 1~2주, 즉시 신청 권고.
5. **Gmail 앱 비밀번호**(사용자 액션): 발신에 쓸 구글 계정에서 2단계 인증 + 앱 비밀번호 발급 → env 전달.

## 7. 리스크

- SMS 심사 리드타임이 일정의 최대 외부 변수 — 미확정 시 S3에서 문자만 뒤로 빼고 메일 우선.
- 상태 노드 추가가 기존 파이프라인 전이와 충돌하지 않도록 TRANSITIONS 확장은 추가-only(개명 금지).
- HTML Storage 전환은 에디터 bridge(이미지 프록시 치환)와 얽혀 있음 — S3에서 회귀 테스트 필수.
